# code-review

请对当前的「工作区的Staged 的 Git 变更」做一次 code review，并按要求输出。不要直接将提示词直接放到文档中, 而是应该按照提示词的内容生成文本

## 你需要完成两件事

### 1. Code Review 文档（Markdown）

输出一份结构化的 code review 文档到/tmp文件夹下,文件命名为code-review-「具体日期时间戳」.md，文件内容包含但不限于：
- **概要**：本次变更的简要总结
- **按文件/模块的评审**：逻辑、可读性、潜在问题、改进建议
- **风险与注意事项**：性能、兼容性、安全等
- **其他建议**：测试、文档、重构建议等

### 2. 推荐的 Commit Message（附在文档末尾）

在整份 code review 文档的**最后**，增加一个二级标题：`## 推荐的 Commit Message`，然后给出一份**带 body 的完整 commit 信息**。

请严格按照以下规则生成 commit message：
+ 格式：必须 feat|fix|refactor|docs|test|chore(scope): subject
+ 语言：subject 用中文, 但是不适合翻译的单词可以用英文
+ 长度：subject ≤ 50 字符
+ 内容：使用简短的语言描述下面的内容
    1. 背景(为什么要改或者bug的现象, 如果是bug则bug的现象留空即可, 用户会手动填入)
    2. 改动(如何改的或者如何修复的)
    3. 影响范围
    4. 相关链接
       - [关联仓库改动链接](填入链接内容)
       - [ui走查单链接](填入链接内容)
       - [bug单链接](填入链接内容)
       - [任务单链接](填入链接内容)
+ 上面提到的细节里面, 不知道的不要编造, 可以留空, 让用户手动填入或者明确提示这部分细节你不清楚,请用户手动填写, 其中相关链接的内容直接原封不动输出即可, 用户会自己删减填写的
