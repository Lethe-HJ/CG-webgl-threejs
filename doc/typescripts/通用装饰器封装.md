# 通用装饰器封装：Angular / NestJS 中类装饰器是如何实现的

在 Angular、NestJS 这类 OOP 很强的框架里，类装饰器主要做三件事：**给类打标记**、**挂/读元数据**、**在需要时替换/包装类**。下面从标准机制到框架用法简要说明。

---

## 1. 类装饰器的标准形式

TypeScript 类装饰器的签名（`experimentalDecorators: true`）是：

```ts
type ClassDecorator = (target: Function) => Function | void;
```

- **target**：被装饰的类的构造函数（即类本身）。
- **返回值**：若返回一个新的 `Function`（构造函数），则运行时会用这个新函数**替换**原类；不返回或返回 `undefined` 则保持原类不变。

```ts
function SimpleDecorator(target: Function) {
  console.log('装饰的类:', target.name);
  // 不返回 = 不替换，只“标记一下”
}

function WrapDecorator<T extends new (...args: any[]) => any>(target: T) {
  return class extends target {
    constructor(...args: any[]) {
      super(...args);
      console.log('实例已创建');
    }
  };
}

@SimpleDecorator
class A {}

@WrapDecorator
class B {}
```

Angular / NestJS 里：有的装饰器只做“标记 + 元数据”（不返回），有的会在内部返回包装类（例如增强生命周期、代理等）。

---

## 2. 元数据与 reflect-metadata：DI 的基础

框架需要知道「这个类的构造函数里各个参数的类型」，才能做**依赖注入**。TypeScript 编译时在开启 `emitDecoratorMetadata: true` 时，会为**被装饰的类**的构造函数参数自动生成元数据，键名为 `'design:paramtypes'`，值为参数类型数组。

这些元数据要**存**起来、**读**出来，用的就是 **reflect-metadata** 库（或等价实现）：

- `Reflect.defineMetadata(key, value, target)`：在类（或成员）上挂元数据。
- `Reflect.getMetadata(key, target)`：从类（或成员）上读元数据。

```ts
import 'reflect-metadata';

function Injectable() {
  return function (target: Function) {
    // 框架在这里可以“什么都不做”，只是标记该类可被注入；
    // 真正的类型信息已经由 TS 在编译时通过 emitDecoratorMetadata 写入了 design:paramtypes
  };
}

class UserService {}
class AppService {
  constructor(private userService: UserService) {}
}

// 开启 emitDecoratorMetadata 后，TS 会为 AppService 生成类似：
// Reflect.defineMetadata('design:paramtypes', [UserService], AppService);
```

容器在创建 `AppService` 时，会 `Reflect.getMetadata('design:paramtypes', AppService)` 得到 `[UserService]`，再递归实例化并注入。这就是 Angular / NestJS 里「构造函数注入」能自动识别类型的原理。

---

## 3. Angular 里类装饰器在做什么

Angular 的类装饰器典型用法：

| 装饰器       | 作用概览 |
|-------------|----------|
| `@Component` | 标记为组件，挂上 selector、template、style 等元数据，并**把该类注册进当前模块的组件表**。 |
| `@Injectable` | 标记为可注入服务，可指定 `providedIn` 等，**注册进注入器**。 |
| `@NgModule`  | 标记为模块，挂 declarations、imports、providers 等，**注册模块与组件/服务关系**。 |
| `@Directive` | 类似组件，标记指令并注册。 |

实现上可以简化为：

1. **打标记**：在类上挂自定义元数据（如 `'angular:component'`、selector、template 等）。
2. **用 reflect-metadata**：读取 `design:paramtypes`（以及可选的 `design:type` 等）供注入器解析依赖。
3. **（可选）包装类**：如需对组件/指令做 AOP（如变更检测、生命周期钩子），可以返回一个继承原类的子类，在子类里包一层逻辑。

```ts
// 概念性简化：Angular 风格的“可注入”标记
function Injectable(metadata?: { providedIn?: 'root' | any }) {
  return function (target: Function) {
    Reflect.defineMetadata('angular:injectable', metadata ?? {}, target);
    // 注入器会 later 用 Reflect.getMetadata('design:paramtypes', target) 解析依赖
  };
}
```

所以：**类装饰器在 Angular 里 = 元数据写入 + 可选的类替换，配合注入器对 `design:paramtypes` 的读取。**

---

## 4. NestJS 里类装饰器在做什么

NestJS 和 Angular 思路接近，也是「类装饰器 + 参数/属性/方法装饰器」配合元数据：

| 装饰器        | 作用概览 |
|---------------|----------|
| `@Controller(path)` | 标记为控制器，把 path 和类挂到路由树。 |
| `@Injectable()`    | 标记为可注入，供模块的 DI 容器使用。 |
| `@Module({...})`    | 标记为模块，挂 imports、controllers、providers 等。 |

同样会：

1. 在类上挂元数据（如 `'path'`、`'controller'`、`'module'`）。
2. 依赖 `reflect-metadata` 的 `design:paramtypes`（以及自定义的 token 元数据）做构造函数注入。
3. 必要时返回包装类（例如给 guard/interceptor 用）。

```ts
// 概念性简化：NestJS 风格的控制器注册
function Controller(path: string) {
  return function (target: Function) {
    Reflect.defineMetadata('path', path, target);
    Reflect.defineMetadata('controller', true, target);
    // 框架在启动时扫描这些元数据，把 target 和 path 注册到路由
  };
}
```

所以：**在 NestJS 里类装饰器同样是“标记 + 元数据”，DI 和路由都依赖这些元数据在启动时做扫描和注册。**

---

## 5. 小结：OOP 框架里类装饰器的实现方式

- **类装饰器**：对**构造函数**做“标记 + 写元数据”，必要时**返回新构造函数**替换原类。
- **类型从哪来**：TypeScript 的 `emitDecoratorMetadata` 为被装饰的类生成 `design:paramtypes` 等，由 **reflect-metadata** 存储，框架的 DI 容器在运行时读取。
- **框架怎么用**：Angular / NestJS 用类装饰器把类**注册**到模块/注入器/路由里，并用参数装饰器（如 `@Body()`、`@Param()`）补充额外元数据；容器在创建实例时根据这些元数据解析依赖并注入。

若要在自己的项目里做“通用装饰器封装”（例如类似本仓库的 [mixin 装饰器](mixin-decorator-usage.md)），可以沿用同一套思路：**类装饰器改/包装类或只写元数据，需要 DI 时配合 reflect-metadata 与 `design:paramtypes`**；不涉及 DI 时，只做“标记 + 可选类替换”即可。
