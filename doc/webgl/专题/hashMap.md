## WebGL中海量数据的存储方案对比

### 场景
在WebGL中通过纹理存储数据量超大（10万×10万）的点，使用线性表还是哈希表？

### 方案一：直接使用index（推荐）
**流程：**
- 点ID → 计算UV坐标 → 纹理采样一次

**特点：**
- 一次纹理fetch
- 无分支、无计算
- GPU纹理单元极度优化
- 空间利用率接近100%

### 方案二：使用hash（不推荐）
**流程：**
- xy → 复杂hash → 生成key → 再去纹理找value

**致命问题：**
1. **hash计算本身就是巨大开销**：乘法、位运算、取模、混合……每个顶点都要跑一遍
2. **hash冲突**：必须处理，一处理就带分支/循环，GPU最怕这个
3. **内存不连续、缓存命中率暴跌**：GPU纹理采样是空间局部性极强的连续访问，hash后完全随机，L1/L2缓存基本失效
4. **空间爆炸**：key不连续意味着需要占据更大的纹理空间，空间利用率可能只有1%～10%

### 性能差距
- **方案一**：60fps轻松，顶点shader极轻
- **方案二**：1～12fps甚至直接卡死，顶点shader极重

**差距：方案二可能比方案一慢5～50倍**

### 最终结论
**只要你能拿到连续index，永远用方案一。**

## 哈希表的本质

### 核心定义
**哈希表 = 线性存储结构 + 哈希映射**

不管上层多花哨，底层一定是一段连续/线性的存储空间，你用key去查，最终一定落到某一个线性位置上。

### 为什么需要哈希？
很多场景下key不是数字（字符串、对象、自定义结构体），没法直接当数组下标用。

### 哈希做了什么？
**把任意类型的key → 映射成一个合法的数组下标**

```
hash(key) → 数组索引
```

### 终极总结
**哈希表 = 给线性存储加了一层"自动算下标"的壳。**
- JS：数组 + 哈希函数 = HashMap
- WebGL：纹理 + uv计算 = 哈希结构

## JS中的哈希表与数组对比

### JS Object/Map 底层就是哈希表
- key会被hash成一个随机位置
- 为了减少冲突，底层数组必须比实际数据大很多
- 空位置也要占内存
- 内存不紧凑、不连续

### 连续数组（Array）是稠密、紧凑、连续
- 内存一块连续
- 没有空洞
- 没有hash计算
- 空间利用率接近100%

### 性能对比

| JS层面 | 内存 | 速度 | 空间利用率 |
|--------|------|------|-----------|
| Array | 紧凑连续 | 极快 | 接近100% |
| Object/Map | 稀疏空洞 | 慢很多 | 50%～70% |

### 访问速度
- **数组[index]**：直接地址 = 基地址 + index × 尺寸，O(1)且极快，CPU缓存命中率极高
- **Map.get(key)/obj[key]**：计算hash → 找位置 → 处理冲突，虽然也是O(1)，但常数大很多，缓存不友好

### 内存占用
- 数组：数据量 ≈ 内存占用
- Map/Object：通常要预留30%～200%空空间，也就是同样数据，内存大1.3～3倍

## 哈希表扩容原理

### 扩容本质
**hash扩容本质就是数组扩容，开辟更大连续内存，迁移数据，再对每个键值重新hash映射成新索引。**

### 具体流程
1. **Hash表扩容 = 底层数组扩容**
   - 新开辟一块更大的连续内存（新数组）
   - 旧数组保留，等迁移完再扔掉

2. **每个元素必须重新算下标（rehash）**
   - 因为容量变了，`hash(key) % newCapacity`结果完全可能变
   - 所以不能简单memcpy复制，必须一个一个重新插

3. **整个过程**
   开新数组 → 遍历旧数据 → rehash → 放到新位置 → 释放旧数组

## 为什么哈希表容量要是2的幂

### 核心原因
**取模运算可以直接换成位与运算，快到极致。**

### 原理解释

#### 正常取模
```
index = hash(key) % capacity
```
取模`%`是除法指令，CPU跑得慢。

#### 当capacity = 2^n时
比如容量 = 16 = 2⁴
- 二进制：`10000`
- 容量-1 = 15
- 二进制：`01111`

这时：
```
hash % 16  ===  hash & 15
```

**取模 = 位与**
**除法 = 门电路级别的快速按位操作**

### 为什么必须是2的幂
- 只有容量是2ⁿ
- `capacity - 1`的二进制才是低位全1
- 才能用`hash & (capacity - 1)`完美代替`hash % capacity`

**CPU做一次&，比做%快一个数量级。**

### 与扩容的关系
容量是2的幂时：
16 → 32 → 64
每次都是**左移一位**
`mask`从`1111` → `11111` → `111111`

rehash极快、极规整。

### 终极总结
**哈希表容量用2的幂，就是为了把慢的取模，变成最快的位与。一切都是为了性能。**

## 哈希冲突处理：链表转红黑树

### 哈希冲突是什么
两个不一样的key，算出来index一样，就叫哈希冲突。

### 解决方案：链地址法
同一个位置挂一条链表：
- 数组下标是头节点
- 冲突的元素往后挂

### 问题：链表太长就废了
- 链表查找：O(n)
- 哈希表本来期望：O(1)

如果一个桶里链表特别长（比如恶意攻击、hash很差），哈希表就退化成链表，慢到炸。

### 优化方案：链表转红黑树
**JDK/Android等源码中的策略：**
- 当链表长度 ≥ 8时，自动转成红黑树
- 当长度 ≤ 6时，退化成链表

### 红黑树是什么
就是一种自平衡的二叉搜索树。
- 查找：O(log n)
- 比链表快得多得多

### 总结
- **数组**：最快，但key不能是字符串/对象
- **哈希**：把任意key转成数组下标
- **链表**：解决冲突，但太长会慢
- **红黑树**：给太长的链表"升级"，保证再怎么冲突也不会退化成O(n)

**链表转红黑树 = 哈希表的兜底保险。**

## 哈希表的内存结构

### 核心结论
**哈希桶里，从来不存在"存值"和"存指针"二选一。它永远、永远、只存一种东西：节点指针/引用。**

### 哈希表底层数组存的是什么
数组里每个格子，存的是**指向一个Node节点的指针（引用）**

```
数组下标：[ 0 ] [ 1 ] [ 2 ] [ 3 ] ...
         指针  指针  指针  指针
```

- 没冲突时：指针指向1个节点
- 有冲突时：指针指向链表的第一个节点

### 冲突时链表怎么"挂"
假设keyA和keyB都算到index=2：

1. 先插keyA
   - 数组[2] = 指向节点A的指针

2. 再插keyB（冲突）
   - 新建节点B
   - 把节点A.next = 节点B
   - 数组[2]还是指向节点A

结构就变成：
```
数组[2] → 节点A → 节点B → ...
```

**"挂"就是靠节点里的next指针连起来。**

### 链表存在哪
**不在数组里！**

- 数组：一段连续内存
- 链表节点：分散在堆的任意位置，每个节点都是独立`new`出来的对象

它们只是通过`next`指针**逻辑上连在一起**，**物理内存完全不连续**。

### 内存结构图示
```
[ 数组 ]（连续内存）
下标0: null
下标1: null
下标2: ║     <- 这里只存指针！
       ║
       ▼
[ 节点A ] (散堆内存)
   keyA valA next → ───→ [ 节点B ] (散堆内存)
                               keyB valB next → ...
```

- 数组：线性连续存储
- 链表：散落各处，靠指针串起来

### 关键理解
**数组只负责"找到是哪条链表"，链表真正存数据，链表不在数组里，是另外开辟的内存，用next指针串起来。**

## 核心知识点总结

1. **WebGL中海量数据存储**：永远用连续index直接纹理采样，不要用hash
2. **哈希表本质**：线性存储结构 + 哈希映射
3. **JS中Array vs Map**：Array更快更省内存，Map/Object是哈希表实现
4. **扩容原理**：开新数组 + rehash + 迁移数据
5. **2的幂容量**：用位与代替取模，性能提升一个数量级
6. **冲突处理**：链表解决冲突，链表太长转红黑树保证性能
7. **内存结构**：数组存指针，链表节点散存在堆中，通过next指针连接

## 实践建议

### WebGL开发
1. 海量数据（10万+级别）永远用连续index纹理存储
2. 避免在shader中进行hash计算
3. 使用分块 + LOD + 视锥体剔除优化

### JS开发
1. 超大数据（百万级）优先使用Array/TypedArray
2. 只有需要字符串key或动态key时才用Object/Map
3. 注意Map/Object的内存开销是数组的1.3-3倍

### 哈希表设计
1. 容量始终保持2的幂
2. 负载因子控制在0.75左右
3. 实现链表转红黑树机制
4. 考虑开放寻址法作为替代方案
