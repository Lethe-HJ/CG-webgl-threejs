我给你把 **Frustum Culling（视锥体剔除）** 讲到**最底层、最工程化、能直接用在你八叉树+点云+海量物体渲染里**的程度，全程细讲、不跳步。

# 一句话核心
**相机看不见的物体，直接不画。**
Frustum Culling = 用数学判断：物体在不在相机视野里。

---

# 0. 先搞懂：什么是 View Frustum？
就是相机的**可视区域**，形状是一个**截顶金字塔**：
- 近裁剪面 near
- 远裁剪面 far
- 视野 FOV
- 宽高比 aspect

这个区域外的物体，**渲染了也看不见**，所以直接剔除。

---

# 1. 原理：用 6 个平面，把空间一切两半
视锥体本质 = **6 个平面**：
- 左 left
- 右 right
- 上 top
- 下 bottom
- 近 near
- 远 far

每个平面把空间分成：
**正面（在里面） / 反面（在外面）**

判断规则：
- **物体完全在任意一个平面的反面 → 剔除，不渲染**
- 否则 → 认为在视锥体内，要渲染

这就是 **视锥体剔除**。

---

# 2. 最关键：怎么判断「物体在不在里面」？
**不用精确判断三角形，只用包围体。**

两种最常用：
1. **Sphere 包围球**（快，不准）
2. **AABB 轴对齐包围盒**（稍慢，更准）

Three.js 默认就是：
**每个 Mesh ⇒ AABB ⇒ 与6个平面做检测**

---

# 3. 平面检测公式（底层原理）
任意平面方程：
\[
ax + by + cz + d = 0
\]

对点 \(p(x,y,z)\)：
\[
dist = a\cdot x + b\cdot y + c\cdot z + d
\]

- dist > 0 → 在平面**正面**（内侧）
- dist < 0 → 在平面**反面**（外侧）
- dist = 0 → 在平面上

### 物体剔除判断：
对 **包围球 / AABB 上的点**：
- **所有点都在某个平面反面 → 剔除**
- 否则 → 保留

---

# 4. Three.js 里默认是怎么工作的？
你平时写的：
```js
mesh.frustumCulled = true; // 默认就是 true
```

内部每帧流程：
1. 相机计算 **6 个视锥平面**
2. 遍历每个 Object3D
3. 取出它的 **worldBoundingBox**（世界坐标系AABB）
4. 对 6 个平面依次判断：盒子是否在平面外
5. **任意一个平面外 → 设为不可见，不进渲染队列**
6. 否则 → 渲染

这就是 Three.js 自带的 **Frustum Culling**。

---

# 5. 重点：为什么你做海量点云/瓦片必须靠它？
假设场景：
- 1000 个点云瓦片
- 相机只能看到 50 个

不剔除：
- 1000 次 DrawCall
- 1000 次状态切换
- CPU 直接爆炸

剔除后：
- **只画 50 个**
- 帧率直接稳 60

**这是海量场景性能第一救命优化。**

---

# 6. 精度与速度对比（实战必知）
### 1）包围球 Sphere
- 极快
- 不准，空隙大
- 适合粒子、大量小球体

### 2）AABB 包围盒
- 快
- 比较准
- Three.js 默认
- 适合瓦片、模型、点云块

### 3）OBB 有向包围盒
- 更准
- 慢
- 一般不用

### 4）八叉树 / 四叉树 + Frustum
- **极快 + 极准**
- 你现在做的就是这个

---

# 7. 你最关心的：八叉树 + Frustum 怎么配合？（工业级流程）
我给你写**最标准的工作流**：

1. 场景构建：
   - 把点云/模型切成瓦片
   - 放进 **八叉树**
   - 每个节点自带 AABB

2. 每帧渲染：
   - 相机生成 6 个视锥平面
   - **遍历八叉树**
   - 如果节点AABB **完全在外面**：跳过整棵子树
   - 如果节点AABB **完全在里面**：渲染所有子节点
   - 如果**相交**：递归往下查
   - 最后得到 **可见物体列表**

3. 渲染：
   - 只渲染可见列表
   - 再做 **状态批处理**
   - 再做 **实例化 / 合并**

这就是 **大型WebGL点云/3D场景** 的标准架构。

# 8. 三个超级重要的坑（你一定会遇到）
## 坑1：动态移动的物体，包围盒没更新
解决方案：
```js
mesh.geometry.computeBoundingBox();
mesh.matrixWorldNeedsUpdate = true;
```

## 坑2：物体被剔除，但其实应该看见
原因：
- AABB 计算不对
- 模型缩放/旋转导致包围盒过期

## 坑3：关闭 frustumCulled 会崩
```js
mesh.frustumCulled = false;
```
只有**全屏、天空盒、后期处理**才关。
普通物体千万别关，关了海量场景必卡。

# 9. 终极总结（记这5句就够）
1. Frustum Culling = **6个平面判断物体是否在相机视野**
2. 看不见 → **直接剔除，不进渲染队列**
3. 用 **AABB/包围球** 快速判断，不精确检测三角形
4. Three.js 默认开启，自动帮你做
5. **八叉树 + Frustum Culling = 海量场景渲染的基石**

好，我们现在讲 **绘制调用优化 / 渲染剔除第 2 条：遮挡剔除（Occlusion Culling）**。

# 一句话核心
**视锥体剔除：扔掉相机看不见的。
遮挡剔除：扔掉相机看得见、但被前面物体挡住的。**

---

# 一、先分清两个剔除（别再搞混）
## 1. Frustum Culling（视锥体剔除）
- 作用：去掉**视野外**的物体
- 开销：极低（数学计算）
- 效果：量大、必开
- 你已经在用：Three.js 默认

## 2. Occlusion Culling（遮挡剔除）
- 作用：去掉**视野内，但被挡住**的物体
- 开销：中高（要渲染/查询）
- 效果：大型室内/建筑/密集场景极强
- **Three.js 没有默认实现，要自己做**

---

# 二、遮挡剔除到底解决什么？
举个你真实场景：
- 你在一栋大楼外面
- 大楼**内部**的所有房间、点云瓦片、模型
  → 都在**视锥体内**
  → 但**完全被墙体挡住**

不做遮挡剔除：
- CPU 要提交所有 DrawCall
- GPU 要绘制所有像素，然后深度测试扔掉
- 造成巨大 **overdraw（过度绘制）**

做遮挡剔除：
- 直接不提交这些物体
- **CPU 压力大减，overdraw 大减**

---

# 三、最核心原理：怎么知道“被挡住了”？
所有遮挡剔除，本质只有一条：

## **先用低开销把“遮挡物”渲染成深度图，
再去测试被遮挡物体是不是全在深度后面。**

主流三种方案：

### 1. 查询法（WebGL 原生 Occlusion Query）
流程：
1. 画**遮挡物（墙、地面、大模型）**的深度
2. 对**潜在被遮挡物体**的包围盒做查询：
   - 渲染它的包围盒
   - 问 GPU：有像素通过深度测试吗？
3. 如果**0 像素通过 = 完全被遮挡 = 剔除**

### 2. 深度贴图法（Depth Texture + CPU 测试）
流程：
1. 先渲染一版**深度纹理**
2. 把物体包围盒投影到屏幕 2D
3. 在 CPU 上采样深度图，比较是否全部被遮挡
4. 是 → 剔除

### 3. 软件遮挡剔除（Software Occlusion Culling）
- 用空间结构（八叉树、BVH）做**遮挡关系推理**
- 完全不依赖 GPU 查询
- 适合大型场景、点云瓦片

---

# 四、WebGL / Three.js 能做哪些？
给你最真实结论：

## ✅ 能做：
1. **软件遮挡剔除（最稳、最推荐）**
2. 基于八叉树的**遮挡查询**
3. 基于深度贴图的**包围盒遮挡测试**

## ❌ 不推荐：
WebGL 的 **occlusionQuery**
- 异步、延迟一帧
- 容易造成卡顿
- 移动端兼容性一般
- **工程上很难用稳**

# 五、你做**点云/瓦片/八叉树**，最实用方案：
## 软件遮挡剔除（Software Occlusion Culling）
我直接给你**可落地的最简流程**：

### 1. 预处理
- 把场景分成**八叉树**
- 每个节点/瓦片有 **AABB 包围盒**
- 标记**遮挡体**：地面、墙、大模型

### 2. 每帧执行
1. 相机位置 → 八叉树找到**当前所在节点**
2. 从相机出发，**从近到远遍历节点**
3. 维护一个**遮挡体AABB列表**
4. 对每个远处瓦片：
   - 判断它的包围盒是否**完全被前面遮挡体包住+挡住**
   - 是 → 标记为遮挡，**不进渲染队列**
5. 剩下的才是**真正可见瓦片**

### 优点：
- 纯 CPU 数学计算
- 无 GPU 同步、无延迟
- 点云/瓦片场景**效果极强**
- 可稳定压掉 30%~70% 无用 DrawCall

# 七、最简单可上手的“低配版遮挡剔除”
你不用做复杂算法，直接用这条就能提升性能：

## **从近到远渲染 + 提前跳过被完全遮挡的瓦片**
规则：
- 瓦片按**距离相机从近到远**排序
- 维护一个**屏幕空间遮挡矩形**
- 如果瓦片投影到屏幕上的区域 **已经完全在遮挡矩形内**
  → 直接剔除

这是**最简单、最有效、最稳**的遮挡剔除入门版。
