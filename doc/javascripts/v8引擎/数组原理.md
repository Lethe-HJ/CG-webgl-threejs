# 一句话结论
V8 里的 JS 数组有**两种完全不同的底层实现**：
1. **快数组（Fast Elements）** —— 连续内存，像真正的数组
2. **慢数组（Slow Elements/Dictionary Elements）** —— 哈希表（字典）

## 1. 快数组（默认）
- 底层是**连续的内存块**
- 索引直接 = 内存偏移
- 访问极快：O(1)
- 自动扩容、缩容
- 适合：**密集、连续**的数组

```js
const arr = [1,2,3,4,5] // 快数组
```

## 2. 慢数组（字典模式）
当数组变得**稀疏**时，V8 会自动降级成慢数组：
- 用 **哈希表** 存：key → value
- 节省内存，但访问变慢
- 适合：**空洞多、索引不连续**

```js
const arr = []
arr[10000] = 1 // 瞬间变慢数组（空洞太多）
```

# 什么时候快数组 ↔ 慢数组 切换？
V8 有一套**阈值判断**：

### 快 → 慢
- 空洞太多，浪费内存 > 阈值
- 索引特别大，导致连续内存装不下

### 慢 → 快
- 数组变得连续、密集
- V8 会重新转成快数组

# 最关键的底层差别（必背）
- **快数组**：连续内存 → 访问快、费内存
- **慢数组**：哈希表 → 访问慢、省内存
- **JS 数组不是传统 C 数组**，是 V8 动态优化的对象

# 补充一个重点：length
JS 数组的 `length` **不是真实元素个数**，而是：
- 快数组：`length` = 容量
- 慢数组：`length` = 最大索引 + 1

这就是为什么：
```js
const arr = []
arr[100] = 1
console.log(arr.length) // 101
```
