要理解 JavaScript 中 Set、Map、WeakMap 的底层原理，核心是抓住 **V8 引擎的存储结构优化** 和 **引用类型的特殊处理**（尤其是 WeakMap）。下面从「底层数据结构」「核心特性实现」「与普通 Object 的区别」三个维度，用新手能懂的方式拆解。

### 一、核心前提：为什么不用普通 Object 代替？
普通 Object 看似是“键值对”，但有天然缺陷：
- 键只能是字符串/符号（Symbol），无法用对象做键；
- 会继承原型链上的属性（如 `toString`），容易冲突；
- 没有原生的“大小统计”“有序性”（ES6 前）；
- 对对象键的引用是强引用，无法被垃圾回收（导致内存泄漏风险）。

Set/Map/WeakMap 正是为解决这些问题而生，底层实现也针对性做了优化。

### 二、Set 和 Map 的底层实现（V8 引擎）
Set（值的集合）和 Map（键值对集合）的底层核心是 **Hash Table（哈希表）**，但 V8 做了关键优化：**采用「哈希表 + 链表」的混合结构（也叫 Hash Table with Separate Chaining）**，同时结合了数组的有序性。

#### 1. 底层存储结构（以 Map 为例，Set 是 Map 的“简化版”）
V8 中 Map 的底层是一个 **HashTable 结构体**，核心包含两部分：
- **数组（Bucket Array）**：数组的每个元素是一个“桶（Bucket）”，桶中存储 `[key, value, next]`（key=键，value=值，next=下一个桶的指针）；
- **链表（解决哈希冲突）**：当两个不同的键计算出相同的哈希值时，会把后插入的键值对挂在当前桶的 `next` 指针上，形成链表。

补充：Set 本质是“值即键”的 Map（`Set.add(x)` 等价于 `Map.set(x, x)`），底层结构和 Map 完全一致，只是只存储值、不存储额外的 value。

#### 2. 核心操作的底层逻辑（以 Map 为例）
##### （1）`map.set(key, value)` 存储逻辑
1. 计算 `key` 的**哈希值**（V8 对不同类型的 key 有专属哈希算法：字符串/数字用常规哈希，对象/数组用“对象标识哈希”）；
2. 用哈希值对桶数组的长度取模，得到该 key 对应的**桶下标**；
3. 检查该桶是否为空：
   - 空：直接将 `[key, value, null]` 存入桶；
   - 非空：遍历桶的链表，检查是否有相同的 key（用 `===` 比较，对象按引用比较）：
     - 有相同 key：更新对应的 value；
     - 无相同 key：新建桶，挂在链表末尾；
4. 若桶数组的“填充率”超过阈值（约 75%），触发**桶数组扩容**（长度翻倍），重新计算所有键的哈希值并迁移。

##### （2）`map.get(key)` 读取逻辑
1. 计算 key 的哈希值，找到对应的桶下标；
2. 遍历该桶的链表，逐个比较 key（`===`）；
3. 找到匹配的 key：返回对应的 value；
4. 没找到：返回 `undefined`。

##### （3）有序性实现
ES6 规定 Map/Set 是**插入有序**的（遍历顺序等于插入顺序），V8 的实现方式是：
- 在哈希表中额外维护一个 **双向链表**，记录所有键值对的插入顺序；
- 遍历（如 `for...of`）时，直接遍历这个双向链表，而非桶数组，保证顺序。

```javascript
// 验证 Map 的有序性和哈希特性
const map = new Map();
const objKey = { id: 1 };

// 插入不同类型的键
map.set('name', '张三');
map.set(objKey, '对象键');
map.set(123, '数字键');

// 遍历顺序与插入顺序一致
for (const [k, v] of map) {
  console.log(k, v); 
  // 输出：
  // name 张三
  // { id: 1 } 对象键
  // 123 数字键
}

// 对象键按引用比较
console.log(map.get({ id: 1 })); // undefined（新对象，引用不同）
console.log(map.get(objKey)); // 对象键（引用相同）
```

#### 3. Map vs Set 底层差异
| 特性         | Map                  | Set                  |
|--------------|----------------------|----------------------|
| 存储内容     | 键值对 `[key, value]` | 仅值（等价于 `[value, value]`） |
| 核心方法     | `set/get/delete`     | `add/has/delete`     |
| 底层结构     | 哈希表 + 双向链表    | 哈希表 + 双向链表（同 Map） |
| 本质         | 键值映射             | 无重复值的集合       |

### 三、WeakMap 的底层实现（核心是“弱引用”）
WeakMap 是 Map 的“特殊版本”，底层结构和 Map 类似（哈希表），但核心差异是 **对“对象键”的引用类型（弱引用）**，这也是它最关键的底层特性。

#### 1. 核心概念：强引用 vs 弱引用
- **强引用（Map/Object）**：如果一个对象被 Map 的 key 引用，即使外部没有其他引用，垃圾回收器（GC）也不会回收这个对象（因为 Map 还“抓着”它）；
- **弱引用（WeakMap）**：WeakMap 的 key 对对象的引用是“弱的”——如果外部没有其他引用指向这个对象，GC 会直接回收该对象，同时自动删除 WeakMap 中对应的键值对。

#### 2. WeakMap 的底层限制（由弱引用决定）
- **key 只能是对象/符号**（不能是字符串、数字等原始类型），否则报错；
- **无法遍历**（没有 `keys()`/`values()`/`entries()`/`for...of`），也没有 `size` 属性——因为 GC 可能随时删除键值对，无法保证遍历结果的准确性；
- **仅支持 `set()`/`get()`/`delete()`/`has()` 四个方法**。

```javascript
// 验证 WeakMap 的弱引用特性
const weakMap = new WeakMap();
let tempObj = { id: 1 };
weakMap.set(tempObj, '测试值');

console.log(weakMap.has(tempObj)); // true
console.log(weakMap.get(tempObj)); // 测试值

// 解除外部引用，触发 GC（浏览器/Node 会自动回收）
tempObj = null;

// 此时 WeakMap 中 { id: 1 } 对应的键值对已被 GC 删除
// 无法直接验证，但可通过内存监控观察（浏览器开发者工具 Memory 面板）
console.log(weakMap.has(tempObj)); // false（tempObj 已为 null）
```

#### 3. WeakMap 的底层应用场景
正因为“自动回收”的特性，WeakMap 常用于：
- 缓存（如存储对象的计算结果，对象销毁后缓存自动清理）；
- DOM 元素的附加数据（DOM 节点被移除后，对应数据自动回收，避免内存泄漏）；
- 私有属性（ES6 类的私有属性底层可基于 WeakMap 实现）。

### 四、与普通 Object 的底层对比
| 特性         | Object              | Map                 | WeakMap             |
|--------------|---------------------|---------------------|---------------------|
| 键类型       | 字符串/符号         | 任意类型（原始+对象）| 仅对象/符号         |
| 引用类型     | 强引用              | 强引用              | 弱引用（对象键）|
| 有序性       | ES6 后插入有序      | 插入有序            | 无（无法遍历）|
| 内存回收     | 手动删除才回收      | 手动删除才回收      | 自动回收（无外部引用）|
| 遍历/大小    | 可遍历，无原生 size | 可遍历，有 size     | 不可遍历，无 size   |

### 总结
1. **Set/Map** 底层基于 **哈希表 + 双向链表** 实现：哈希表保证快速增删查（O(1) 时间复杂度），双向链表保证插入有序，Set 是“值即键”的简化版 Map；
2. **WeakMap** 底层仍是哈希表，但核心是对对象键的**弱引用**：GC 会自动回收无外部引用的对象键，因此无法遍历、无 size 属性；
3. 三者相比普通 Object，解决了“键类型限制”“强引用内存泄漏”“有序性”等问题，是不同场景下更专业的键值存储方案。
